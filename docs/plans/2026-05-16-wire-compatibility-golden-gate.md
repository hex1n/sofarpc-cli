# Wire Compatibility Golden Gate

## 1. Problem Frame

The pure-Go invoke path now sends BOLT frames and Hessian content without a Java
sidecar. Existing tests prove internal consistency, but they do not yet prove
wire compatibility with the real Java SOFARPC/Hessian runtime.

Current gaps:

- `internal/sofarpcwire` has Go round-trip tests, but Go-generated response
  content can accidentally encode a shape Java would never produce.
- The golden fixture harness already exists, but it skips when no fixture is
  checked in.
- Request encoding is not verified by a Java decoder.
- Response decoding is not backed by Java-produced Hessian content.
- CI does not distinguish fast Go regression checks from heavier Java fixture
  regeneration.

The goal is to make wire compatibility a release gate while keeping ordinary Go
development fast.

## 2. Scope

In scope:

- Extend the golden fixture schema to support both request and response content.
- Make Go golden tests fail when required committed fixtures are missing.
- Add first-phase `com.example.*` fixtures for common SOFARPC generic invoke
  shapes.
- Add a Java fixture generator/verifier using pinned real SOFARPC/Hessian
  dependencies.
- Add a manual CI workflow for Java fixture regeneration and consistency checks.

Out of scope for the first phase:

- Full Java provider/client network e2e as part of default CI.
- Registry invoke support.
- Sanitized real-business fixtures.
- Replacing the existing BOLT frame unit tests.
- Moving Java generation into the default push/PR workflow.

## 3. Fixture Schema

All fixtures live under:

```text
internal/sofarpcwire/testdata/golden/
```

Top-level fields:

```json
{
  "name": "dto-bigdecimal-list",
  "description": "short human-readable purpose",
  "kind": "request-content",
  "contentHex": "4f...",
  "want": {}
}
```

Rules:

- `kind` is either `request-content` or `response-content`.
- `contentHex` is Hessian content only, not a BOLT frame.
- `want` stores semantic expectations, not byte-for-byte internal details.
- Unknown fields may be rejected to keep fixture drift visible.
- Committed fixtures come from one deterministic baseline SOFARPC version.
  Additional company/project SOFARPC versions are verified in a manual matrix,
  not committed as parallel copies of every fixture.

### 3.1 Request Fixtures

`request-content` proves Java can decode Go-produced request content.

Expected `want` fields:

```json
{
  "service": "com.example.FixtureFacade",
  "method": "query",
  "paramTypes": ["com.example.FixtureRequest"],
  "targetServiceUniqueName": "com.example.FixtureFacade:1.0",
  "argsJson": [
    {
      "type": "com.example.FixtureRequest",
      "fields": {
        "id": 1001
      }
    }
  ]
}
```

The Java verifier owns this direction. The Go test only validates schema and
coverage for request fixtures unless a small Go-side semantic assertion is
useful.

### 3.2 Response Fixtures

`response-content` proves Go can decode Java-produced response content.

Expected `want` fields:

```json
{
  "isError": false,
  "errorMsg": "",
  "appResponseType": "com.example.FixtureResult",
  "appResponseJson": {
    "type": "com.example.FixtureResult",
    "fields": {
      "success": true
    }
  },
  "responseProps": {}
}
```

The Go test decodes `contentHex` with `DecodeResponse` and asserts `want`.
`appResponseJson` is matched as a semantic subset of the decoded tree.

## 4. First Fixture Set

Create a small Java fixture domain under:

```text
internal/sofarpcwire/testdata/java-fixtures/
```

First-phase fixture shapes:

- scalar request/response: `String`, `int`, `long`, `boolean`
- DTO request/response with ordinary fields
- `BigDecimal` and `BigInteger`
- `List<DTO>`
- `Map<String, DTO>`
- nested DTO
- enum value
- remote exception or SofaResponse error path

Use `com.example.*` names only. Do not include project-specific service names,
hosts, registry addresses, or business payloads.

## 5. Java Fixture Project

Location:

```text
internal/sofarpcwire/testdata/java-fixtures/
```

Responsibilities:

- Use pinned Maven dependencies for the real Java SOFARPC/Hessian runtime.
- Allow the SOFARPC dependency version to be selected with
  `-Dsofarpc.version=...`; default to the repository's current baseline.
- Generate `response-content` fixtures from Java-encoded `SofaResponse`
  payloads.
- Verify `request-content` fixtures generated by Go by decoding them with the
  Java runtime and comparing semantic expectations.
- Provide one reproducible command for regeneration.
- Avoid network or live SOFARPC provider requirements.

The Java project is not imported by Go packages because it lives under
`testdata`.

## 6. Go Test Changes

Update `internal/sofarpcwire/golden_fixture_test.go`:

- fail when no golden fixture JSON exists
- fail when no `request-content` fixture exists
- fail when no `response-content` fixture exists
- fail on unknown `kind`
- fail on empty `contentHex`
- keep response decoding assertions in Go
- validate request fixture schema even when Java verifier is not running

Add helper validation for fixture shape so fixture errors are reported before
decode errors.

## 7. CI

Default `ci.yml`:

- keep `go vet ./...`
- keep `go test -race ./...`
- keep direct BOLT e2e smoke
- rely on Go golden tests to consume committed fixture JSON

Manual workflow:

- run Java fixture regeneration/verifier for the baseline version
- fail if generated fixture output differs from committed
  `internal/sofarpcwire/testdata/golden/*.json`
- run the Java verifier and Go decoder tests against the declared SOFARPC
  version matrix using temporary generated fixture directories
- document this workflow as required before release

The manual workflow can extend the existing `e2e.yml` or become a separate
`wire-fixtures.yml`.

## 8. Implementation Units

### Unit A: Expand Go Fixture Harness

Files:

- `internal/sofarpcwire/golden_fixture_test.go`
- `internal/sofarpcwire/testdata/golden/README.md`

Acceptance:

- no fixtures means test failure
- fixture with unknown `kind` fails
- request and response coverage are both required
- existing response assertions still pass for response fixtures

### Unit B: Add Initial Fixture JSON

Files:

- `internal/sofarpcwire/testdata/golden/*.json`

Acceptance:

- at least one `request-content` fixture
- at least one `response-content` fixture
- first batch covers DTO, decimal, container, nested DTO, enum, and error path
  either as separate fixtures or a compact combined set

Implemented baseline fixtures:

- `request-query-fixture.json`
- `response-success-dto.json`
- `response-error.json`

### Unit C: Add Java Fixture Generator/Verifier

Files:

- `internal/sofarpcwire/testdata/java-fixtures/pom.xml`
- `internal/sofarpcwire/testdata/java-fixtures/src/...`

Acceptance:

- Java command regenerates deterministic fixture JSON
- Java verifier decodes `request-content` and checks `want`
- Java generator writes `response-content`
- dependency versions are pinned
- SOFARPC version can be switched with `-Dsofarpc.version=...`

### Unit D: Add Manual CI Workflow

Files:

- `.github/workflows/wire-fixtures.yml`

Acceptance:

- workflow can be run with `workflow_dispatch`
- workflow runs the Java fixture command
- workflow fails when generated fixtures differ from committed fixtures

Implemented:

- baseline job regenerates committed fixtures and checks git drift
- matrix job verifies SOFARPC `5.4.0`, `5.7.6`, and `5.8.0` using temporary
  fixture directories

### Unit E: Document Release Requirement

Files:

- `README.md`
- `README.zh-CN.md`

Acceptance:

- release instructions mention the Java fixture workflow
- ordinary install docs remain Go-only

Implemented:

- README release checklist requires the manual `wire-fixtures` workflow
- Chinese README documents the same Go-only default and release gate split

## 9. Open Follow-ups

- Add sanitized real-project fixtures after the fake fixture domain is stable.
- Consider nightly Java fixture regeneration if manual workflow proves reliable
  and fast.
- Consider full Java provider/client network e2e as a separate release check,
  not as part of the first golden gate.
