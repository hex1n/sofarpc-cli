# rpcctl 0.4 Implementation Plan

## Goal

Upgrade `rpcctl` from a working portable CLI into a tool that stays reliable on real SOFABoot / SOFARPC projects.

## Phase 0.4.0-core

Status: complete

1. Overload-aware manifest and metadata
  - Keep the existing single-method shape compatible.
  - Add `overloads[]` support for methods with multiple signatures.
  - Resolve overloads by `--types` first, then by argument count when that is unambiguous.
   - Fail fast when metadata is ambiguous instead of guessing.

2. Structured runtime error protocol
   - Return stable runtime-side error fields instead of relying on message parsing.
   - Keep launcher hints as a presentation layer, not the source of truth.

3. Runtime version matrix
   - Add explicit supported runtime manifests.
   - Show detected version source and fallback behavior in diagnostics.

## Phase 0.4.1-ops

Status: complete

1. `rpcctl doctor`
  - Check Java, manifest/context discovery, runtime resolution, target reachability, and version inference.

2. Hardened download and install flow
   - Add archive verification.
   - Add clearer verbose diagnostics for download failures.
   - Add concurrency-safe runtime cache behavior.

3. Focused automated tests
   - Cover version detection, type normalization, overload resolution, and runtime marker parsing.

## Phase 0.4.2-internal

Status: complete

1. Split launcher responsibilities
   - Extract shared environment/runtime/stub resolution out of `RpcCtlApplication`.
   - Extract shared context discovery and doctor report aggregation out of `RpcCtlApplication`.
   - Move `invoke` invocation/result annotation and failure-hint assembly out of `RpcCtlApplication`.
   - Move `doctor` report assembly and network/runtime section construction out of `RpcCtlApplication`.

2. Simplify CI/build flow
   - Remove duplicate build paths between CI and smoke scripts.

## Acceptance Criteria

- Overloaded Java interfaces can be imported without dropping methods.
- Ambiguous overloads require `--types` instead of silently guessing.
- Distinct-arity overloads can be selected from metadata without `--types`.
- The repository keeps a repeatable verification path for direct, registry, stub-aware, and overload-aware calls.
